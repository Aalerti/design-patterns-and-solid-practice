# Good Solution: Liskov Substitution Principle

В этой версии код исправлен так, чтобы он лучше соблюдал Liskov Substitution Principle.

## Что было не так в bad-версии

В плохом варианте базовый класс `Account` заставлял все счета реализовывать метод `withdraw()`.

Для `DebitAccount` и `CreditAccount` это нормально: с таких счетов можно снять деньги. Но `DepositAccount` тоже наследовался от `Account` и был вынужден реализовать `withdraw()`, хотя вклад нельзя использовать как обычный счет для снятия через банкомат.

Из-за этого `DepositAccount.withdraw()` выбрасывал `UnsupportedOperationException`. Это нарушение LSP: объект дочернего класса нельзя было безопасно использовать там, где ожидается родительский `Account` с рабочим снятием денег.

## Что исправлено

- Общие свойства счета остались в `Account`: владелец и баланс.
- Возможность снятия денег вынесена в отдельный интерфейс `WithdrawableAccount`.
- `DebitAccount` и `CreditAccount` реализуют `WithdrawableAccount`, потому что с них можно снять деньги.
- `DepositAccount` остается наследником `Account`, но больше не обязан иметь метод `withdraw()`.
- `ATM` теперь принимает `List<WithdrawableAccount>`, а не `List<Account>`.

## Почему теперь это LSP

`DepositAccount` больше не притворяется счетом, который поддерживает снятие денег. Он остается банковским счетом, поэтому его можно использовать там, где нужен обычный `Account`: например, показать владельца или баланс.

Но его нельзя передать в `ATM`, потому что `ATM` работает только с контрактом `WithdrawableAccount`. Ошибка теперь предотвращается на этапе компиляции, а не падает в рантайме через `UnsupportedOperationException`.

Главная идея: код зависит от того контракта, который ему реально нужен. Банкомату нужен не любой счет, а только счет, поддерживающий снятие денег.
