# 4.2.4 SOLID: ISP — Interface Segregation Principle

> **ISP**: клиенты не должны зависеть от методов, которые они не используют.

## Короткая идея

**ISP** расшифровывается как **Interface Segregation Principle**, или **принцип разделения интерфейсов**.

Простыми словами:

> Лучше несколько маленьких, точных интерфейсов, чем один большой интерфейс “на все случаи жизни”.

Если класс реализует интерфейс, он должен реализовывать только те методы, которые ему действительно нужны. Если интерфейс заставляет класс писать пустые методы, кидать `UnsupportedOperationException` или притворяться, что он умеет больше, чем на самом деле, это сигнал нарушения ISP.

## Почему этот принцип существует

Представь интерфейс:

```java
interface Worker {
    void work();
    void eat();
    void sleep();
}
```

Для человека это может быть нормально:

```java
class HumanWorker implements Worker {
    public void work() {
        // working
    }

    public void eat() {
        // eating
    }

    public void sleep() {
        // sleeping
    }
}
```

Но что делать с роботом?

```java
class RobotWorker implements Worker {
    public void work() {
        // working
    }

    public void eat() {
        throw new UnsupportedOperationException();
    }

    public void sleep() {
        throw new UnsupportedOperationException();
    }
}
```

Формально код компилируется. Архитектурно — плохо.

Робота заставили зависеть от методов, которые к нему не относятся. Интерфейс оказался слишком толстым.

Такой интерфейс часто называют:

```text
fat interface
```

или:

```text
god interface
```

## Главная проблема больших интерфейсов

Большой интерфейс создаёт ложную обязанность.

Класс как будто говорит:

```text
Я умею всё, что описано в интерфейсе.
```

Но на деле часть методов ему не подходит.

Тогда появляются плохие реализации:

```java
public void print() {
    // do nothing
}
```

или:

```java
public void scan() {
    throw new UnsupportedOperationException();
}
```

или:

```java
public void fax() {
    throw new IllegalStateException("Not supported");
}
```

Это не просто некрасиво. Это опасно, потому что вызывающий код видит интерфейс и ожидает, что все методы работают.

## Схема нарушения ISP

```text
Большой интерфейс
        |
        v
Класс обязан реализовать все методы
        |
        v
Часть методов классу не подходит
        |
        v
Пустые реализации / исключения / странная логика
        |
        v
Клиент зависит от лишнего поведения
        |
        v
ISP нарушен
```

## Правильная идея

Вместо одного большого интерфейса:

```text
Worker
 ├─ work()
 ├─ eat()
 └─ sleep()
```

лучше сделать несколько маленьких интерфейсов:

```text
Workable
 └─ work()

Eatable
 └─ eat()

Sleepable
 └─ sleep()
```

Тогда каждый класс реализует только то, что ему действительно подходит.

## Исправленный пример

Разделяем интерфейсы по способностям:

```java
interface Workable {
    void work();
}

interface Eatable {
    void eat();
}

interface Sleepable {
    void sleep();
}
```

Человек может реализовать все три:

```java
class HumanWorker implements Workable, Eatable, Sleepable {
    public void work() {
        // working
    }

    public void eat() {
        // eating
    }

    public void sleep() {
        // sleeping
    }
}
```

Робот реализует только рабочее поведение:

```java
class RobotWorker implements Workable {
    public void work() {
        // working
    }
}
```

Теперь робот не обязан притворяться, что он ест или спит.

Код, которому нужен работник, принимает `Workable`:

```java
void startShift(Workable worker) {
    worker.work();
}
```

Код, которому нужен кто-то, кто может есть, принимает `Eatable`:

```java
void lunchBreak(Eatable eater) {
    eater.eat();
}
```

Такой дизайн честнее и безопаснее.

## Важная мысль

ISP не говорит:

```text
Каждый интерфейс должен иметь только один метод.
```

Он говорит:

```text
Интерфейс должен описывать цельную роль.
```

Если методы естественно используются вместе, они могут быть в одном интерфейсе.

Например:

```java
interface Repository<T> {
    T findById(long id);
    void save(T value);
    void deleteById(long id);
}
```

Такой интерфейс может быть нормальным, если в системе действительно есть роль “репозиторий”, и все реализации обязаны поддерживать эти операции.

Проблема начинается, когда в интерфейс добавляют всё подряд:

```java
interface Repository<T> {
    T findById(long id);
    void save(T value);
    void deleteById(long id);
    void exportToPdf();
    void sendEmailReport();
    void rebuildSearchIndex();
    void clearCache();
}
```

Здесь смешались разные ответственности:

```text
хранение данных
экспорт
email
поисковый индекс
кеш
```

Такой интерфейс будет болезненно реализовывать и тяжело поддерживать.

## ISP и клиенты

В названии принципа важны не только интерфейсы, но и **клиенты**.

Клиент — это код, который использует интерфейс.

Например:

```java
void printDocument(MultifunctionDevice device) {
    device.print();
}
```

Если метод использует только `print()`, ему не нужен интерфейс, в котором есть ещё `scan()` и `fax()`.

Плохо:

```java
interface MultifunctionDevice {
    void print();
    void scan();
    void fax();
}
```

Лучше:

```java
interface Printer {
    void print();
}

interface Scanner {
    void scan();
}

interface Fax {
    void fax();
}
```

Теперь клиент зависит только от того, что реально использует:

```java
void printDocument(Printer printer) {
    printer.print();
}
```

Это уменьшает связанность.

## Что такое связанность

**Связанность** — это насколько сильно один кусок кода зависит от другого.

Если код зависит от большого интерфейса, он связан со всеми методами этого интерфейса, даже если использует только один.

Например:

```text
Метод использует только print(),
но зависит от MultifunctionDevice,
где есть print(), scan(), fax().
```

Значит, изменение `scan()` или `fax()` может затронуть код, которому они вообще не нужны.

ISP помогает сделать зависимости точнее:

```text
Нужна печать -> зависим от Printer.
Нужно сканирование -> зависим от Scanner.
Нужна отправка факса -> зависим от Fax.
```

## ISP и изменение кода

Большой интерфейс опасен тем, что любое изменение в нём бьёт по всем реализациям.

Допустим, был интерфейс:

```java
interface Device {
    void print();
}
```

Его реализуют десять классов.

Потом в него добавили:

```java
void scan();
```

Теперь все десять классов должны измениться, даже если сканировать умеют только два.

Это нарушение ISP, потому что изменение одной роли заставило меняться классы, которым эта роль не нужна.

## ISP и LSP

ISP тесно связан с LSP.

Если интерфейс слишком большой, классы начинают реализовывать неподходящие методы через:

```java
throw new UnsupportedOperationException();
```

А это уже может привести к нарушению LSP.

Почему?

Потому что объект формально реализует интерфейс, но на самом деле не поддерживает часть его поведения.

```text
ISP помогает не создавать такие ложные контракты.
LSP помогает проверять, можно ли безопасно подставлять реализацию.
```

Коротко:

```text
ISP: не заставляй класс зависеть от лишних методов.
LSP: если класс обещал контракт, он должен его выполнять.
```

## ISP и SRP

ISP также связан с SRP.

**SRP** говорит:

```text
У класса должна быть одна причина для изменения.
```

**ISP** говорит:

```text
У интерфейса должна быть точная роль, а клиенты не должны зависеть от лишнего.
```

Если интерфейс смешивает несколько ролей, у него появляется несколько причин для изменения.

Например:

```text
ReportService
 ├─ generateReport()
 ├─ saveReport()
 ├─ sendReportByEmail()
 └─ printReport()
```

Здесь могут быть разные причины для изменения:

```text
изменилась генерация отчёта
изменилась база данных
изменилась отправка email
изменилась печать
```

Часто это значит, что интерфейс стоит разделить.

## Типичные признаки нарушения ISP

### 1. Пустые методы

```java
public void scan() {
    // not needed
}
```

Класс вынужден реализовать метод, который ему не нужен.

### 2. UnsupportedOperationException

```java
public void fax() {
    throw new UnsupportedOperationException();
}
```

Класс формально реализует контракт, но фактически отказывается его выполнять.

### 3. Интерфейс называется слишком общо

Подозрительные названия:

```text
Manager
Service
Handler
Processor
Helper
Util
```

Такие названия не всегда плохие, но часто за ними скрывается слишком много обязанностей.

### 4. Реализации используют только часть интерфейса

Если одна реализация использует методы `a`, `b`, `c`, а другая — только `a`, а третья — только `b`, это сигнал.

Возможно, интерфейс описывает не одну роль, а несколько.

### 5. Изменение интерфейса ломает много несвязанных классов

Если добавление одного метода заставляет менять множество классов, которым этот метод не нужен, интерфейс слишком широкий.

## Как применять ISP на практике

Перед созданием интерфейса полезно спросить себя:

```text
1. Кто будет клиентом этого интерфейса?
2. Какие методы клиент действительно вызывает?
3. Эти методы относятся к одной роли?
4. Все реализации смогут честно выполнить каждый метод?
5. Не появятся ли пустые методы или UnsupportedOperationException?
6. Можно ли назвать интерфейс конкретнее?
```

Если интерфейс трудно назвать одним точным словом, возможно, он делает слишком много.

## Хорошее правило именования

Интерфейс часто лучше называть по роли или способности:

```text
Readable
Writable
Closeable
Comparable
Runnable
Iterable
Printer
Scanner
Repository
Validator
Formatter
```

Хорошее имя помогает увидеть границы ответственности.

Если хочется назвать интерфейс:

```text
EverythingService
UniversalManager
CommonProcessor
```

это повод остановиться и подумать.

## Пример с устройствами

Плохой вариант:

```java
interface Machine {
    void print();
    void scan();
    void fax();
}
```

Обычный принтер не умеет сканировать и отправлять факсы:

```java
class SimplePrinter implements Machine {
    public void print() {
        // printing
    }

    public void scan() {
        throw new UnsupportedOperationException();
    }

    public void fax() {
        throw new UnsupportedOperationException();
    }
}
```

Лучший вариант:

```java
interface Printer {
    void print();
}

interface Scanner {
    void scan();
}

interface FaxMachine {
    void fax();
}
```

Теперь простой принтер реализует только печать:

```java
class SimplePrinter implements Printer {
    public void print() {
        // printing
    }
}
```

А многофункциональное устройство может реализовать несколько интерфейсов:

```java
class OfficeMachine implements Printer, Scanner, FaxMachine {
    public void print() {
        // printing
    }

    public void scan() {
        // scanning
    }

    public void fax() {
        // sending fax
    }
}
```

## Важное отличие: маленький интерфейс не всегда хороший

Не надо механически дробить всё до одного метода.

Плохо, если из-за чрезмерного дробления код становится шумным:

```java
interface NameReadable {
    String name();
}

interface AgeReadable {
    int age();
}

interface EmailReadable {
    String email();
}
```

Если эти данные всегда используются вместе как единая роль, такое разделение может быть лишним.

ISP — не про минимальный размер. ISP — про точность зависимости.

Главный вопрос:

```text
Клиенту действительно нужен весь этот интерфейс?
```

## Где ISP встречается в Java

В стандартной библиотеке Java есть много хороших маленьких интерфейсов:

```java
Runnable
Comparable<T>
Iterable<T>
Iterator<T>
AutoCloseable
Closeable
Serializable
Cloneable
```

Например, `Runnable` описывает одну роль:

```java
void run();
```

`Comparable<T>` тоже описывает одну роль:

```java
int compareTo(T other);
```

`Iterable<T>` говорит:

```java
Iterator<T> iterator();
```

Такие интерфейсы легко понимать, реализовывать и использовать.

## ISP в Spring

В Spring принцип ISP тоже очень важен.

Лучше зависеть от узкого интерфейса:

```java
interface EmailSender {
    void sendEmail(String address, String message);
}
```

чем от большого сервиса, который умеет всё:

```java
interface NotificationService {
    void sendEmail(String address, String message);
    void sendSms(String phone, String message);
    void sendPush(String userId, String message);
    void generateReport();
    void clearQueue();
}
```

Если классу нужна только отправка email, он должен зависеть от `EmailSender`, а не от огромного `NotificationService`.

Это облегчает:

```text
тестирование
замену реализации
чтение кода
поддержку
мокирование в unit-тестах
```

## Связь с тестированием

ISP делает тесты проще.

Если класс зависит от большого интерфейса, в тесте приходится мокать большой контракт, даже если нужен один метод.

Плохо:

```text
Тестируем класс, которому нужен только print(),
но вынуждены создавать mock для print(), scan(), fax(), connect(), disconnect().
```

Хорошо:

```text
Класс зависит от Printer.
В тесте мокается только Printer.
```

Чем точнее зависимость, тем проще тест.

## Краткое сравнение: до и после ISP

До ISP:

```text
Один большой интерфейс
Много лишних методов
Пустые реализации
UnsupportedOperationException
Сильная связанность
Сложные тесты
Изменения задевают много классов
```

После ISP:

```text
Несколько точных интерфейсов
Классы реализуют только нужное
Контракты честные
Связанность ниже
Тесты проще
Изменения локальнее
Код легче читать
```

## Алгоритм применения ISP

```text
1. Найди большой интерфейс.
2. Посмотри, какие клиенты какие методы реально используют.
3. Сгруппируй методы по ролям.
4. Вынеси роли в отдельные интерфейсы.
5. Пусть классы реализуют только подходящие интерфейсы.
6. Пусть клиенты зависят от самого узкого подходящего интерфейса.
```

## Мини-шпаргалка

```text
ISP = не заставляй клиента зависеть от лишнего.

Плохой сигнал:
интерфейс большой, а реализации используют только его часть.

Ещё плохой сигнал:
в реализации есть пустые методы или UnsupportedOperationException.

Хорошее решение:
разделить интерфейс на маленькие роли.
```

## Главное, что нужно запомнить

> ISP — это принцип про честные и точные зависимости.

Интерфейс должен описывать не “всё, что когда-нибудь может понадобиться”, а конкретную роль, которую клиент действительно использует.

Хороший интерфейс похож на ясный договор:

```text
Я обещаю только то, что действительно умею.
Ты зависишь только от того, что действительно используешь.
```
