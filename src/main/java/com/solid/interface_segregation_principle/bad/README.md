# SOLID Practice: Interface Segregation Principle

Твоя задача - исправить код так, чтобы он лучше соблюдал Interface Segregation Principle.

> Клиенты не должны зависеть от методов, которые они не используют.

## Что сейчас не так

В `Main.java` есть интерфейс `OfficeDevice`, в котором сразу три метода:

- `print()`;
- `scan()`;
- `fax()`.

`MultiFunctionPrinter` действительно умеет печатать, сканировать и отправлять факс, поэтому для него такой интерфейс подходит.

Но `SimplePrinter` умеет только печатать. Из-за большого интерфейса он вынужден реализовывать `scan()` и `fax()`, хотя эти операции ему не подходят. Поэтому внутри этих методов выбрасывается `UnsupportedOperationException`.

Это нарушение ISP: класс зависит от лишних методов и вынужден притворяться, что поддерживает возможности, которых у него нет.

## Задание

Исправь структуру интерфейсов так, чтобы:

- `SimplePrinter` больше не был обязан реализовывать `scan()` и `fax()`;
- печатающие устройства работали через отдельный маленький интерфейс;
- сканирующие устройства работали через отдельный маленький интерфейс;
- `ReportService` принимал только тот интерфейс, который нужен конкретному методу;
- программа не падала с `UnsupportedOperationException`.

## Подсказка

Можно разделить большой интерфейс на несколько маленьких:

```java
interface Printable {
    void print(String document);
}

interface Scannable {
    void scan(String document);
}

interface Faxable {
    void fax(String document);
}
```

Тогда `SimplePrinter` может быть только `Printable`, а `MultiFunctionPrinter` может реализовать все нужные интерфейсы.

## Самопроверка

После решения спроси себя:

- Есть ли класс, который реализует метод через `UnsupportedOperationException`?
- Может ли метод печати принимать только устройства, которые умеют печатать?
- Может ли метод сканирования принимать только устройства, которые умеют сканировать?
- Остался ли интерфейс, который заставляет класс зависеть от лишних методов?
