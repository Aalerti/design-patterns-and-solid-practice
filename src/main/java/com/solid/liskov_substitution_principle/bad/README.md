# SOLID Practice: Liskov Substitution Principle

Твоя задача - исправить код так, чтобы он лучше соблюдал Liskov Substitution Principle.

> Объекты дочернего класса должны заменять объекты родительского класса без поломки программы.

## Что сейчас не так

В `Main.java` есть базовый класс `Account`, у которого есть обязательный метод `withdraw()`.

`DebitAccount` и `CreditAccount` действительно позволяют снять деньги, поэтому для них `withdraw()` выглядит нормально.

Но `DepositAccount` тоже наследуется от `Account`, хотя вклад нельзя использовать как обычный счет для снятия денег через банкомат. Из-за этого `DepositAccount.withdraw()` выбрасывает `UnsupportedOperationException`.

Получается проблема: `ATM` ожидает, что любой `Account` поддерживает снятие денег, но один наследник ломает это ожидание. Это нарушение LSP.

## Задание

Исправь структуру классов так, чтобы:

- `DepositAccount` остался банковским счетом;
- `DepositAccount` больше не был обязан реализовывать `withdraw()`;
- `ATM` мог работать только со счетами, с которых действительно можно снять деньги;
- программа не падала с `UnsupportedOperationException`.

## Подсказка

Можно разделить общие свойства счета и возможность снятия денег.

Например:

```java
abstract class Account {
    // owner, balance и общая логика счетов
}

interface Withdrawable {
    void withdraw(double amount);
}
```

Тогда `DebitAccount` и `CreditAccount` могут быть `Withdrawable`, а `DepositAccount` - нет.

## Самопроверка

После решения спроси себя:

- Можно ли передать `DepositAccount` туда, где ожидают любой банковский счет?
- Можно ли случайно передать `DepositAccount` в `ATM` для снятия денег?
- Остался ли где-то метод, который наследник вынужден реализовывать через исключение?
