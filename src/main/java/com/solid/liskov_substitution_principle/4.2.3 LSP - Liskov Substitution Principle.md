# 4.2.3 SOLID: LSP — Liskov Substitution Principle

## Что такое LSP

**LSP** расшифровывается как **Liskov Substitution Principle**, или **принцип подстановки Барбары Лисков**.

Простая формулировка:

> Если класс `Child` наследуется от класса `Parent`, то объект `Child` должен спокойно использоваться везде, где ожидается `Parent`, и программа не должна ломаться логически.

То есть наследник должен быть настоящей разновидностью родителя, а не просто похожим классом, у которого удобно переиспользовать поля и методы.

## Идея на пальцах

Представим, что есть код, который работает с `Bird`:

```java
void makeBirdFly(Bird bird) {
    bird.fly();
}
```

Если передать туда `Sparrow`, всё нормально: воробей — птица и умеет летать.

Но если сделать `Penguin extends Bird`, а внутри `fly()` написать:

```java
throw new UnsupportedOperationException();
```

то Java формально разрешит такое наследование, но с точки зрения дизайна это нарушение LSP.

Код ожидал:

```text
Мне дали Bird, значит я могу вызвать fly().
```

А наследник внезапно говорит:

```text
Нет, я так не умею.
```

Главная мысль: **LSP не про синтаксис Java, а про контракт поведения**.

Компилятор проверяет только:

```text
Penguin is a Bird
```

А LSP спрашивает:

```text
Можно ли Penguin безопасно использовать как Bird во всех сценариях?
```

Если нет, наследование выбрано неправильно.

## Контракт класса

Контракт класса — это не только сигнатуры методов. Это ещё и ожидания от поведения.

Например, если у родителя есть метод `fly()`, его контракт может быть таким:

```text
Метод fly():
- должен завершаться без UnsupportedOperationException
- должен переводить объект в состояние "летит"
- не должен ломать инварианты объекта
```

**Инвариант** — это правило, которое всегда должно быть истинным для объекта.

Примеры инвариантов:

```text
ширина прямоугольника неотрицательная
баланс счёта не может стать меньше нуля
закрытая дверь не может одновременно быть открытой
```

LSP говорит: наследник не должен нарушать контракт родителя.

## Три правила LSP

Есть три важных правила:

```text
1. Наследник не должен требовать больше, чем родитель.
2. Наследник не должен обещать меньше, чем родитель.
3. Наследник не должен ломать инварианты родителя.
```

В терминах проектирования это часто формулируют так:

```text
Preconditions нельзя усиливать.
Postconditions нельзя ослаблять.
Invariants нельзя нарушать.
```

По-человечески:

```text
Если Parent принимал любое число,
Child не должен внезапно принимать только положительные.

Если Parent обещал вернуть отсортированный список,
Child не должен вернуть случайный порядок.

Если Parent гарантировал валидное состояние,
Child не должен после вызова оставлять объект в сломанном состоянии.
```

## Схема проверки LSP

```text
Код работает с Parent
        |
        v
Вызывает методы Parent
        |
        v
Подставляем Child
        |
        v
Поведение всё ещё корректное?
        |
   да ------> LSP соблюдён
   нет -----> наследование подозрительное
```

Главный тест LSP:

```text
Если заменить объект родителя на объект наследника,
потребуется ли менять вызывающий код?
```

Если потребуется проверка вида:

```java
if (object instanceof SomeChild) {
    // особый случай
}
```

то это сигнал, что иерархия может нарушать LSP.

## Пример нарушения: Rectangle и Square

Классический пример — прямоугольник и квадрат.

Математически квадрат — это прямоугольник. Но в ООП это не всегда хорошее наследование.

```java
class Rectangle {
    protected int width;
    protected int height;

    void setWidth(int width) {
        this.width = width;
    }

    void setHeight(int height) {
        this.height = height;
    }

    int area() {
        return width * height;
    }
}
```

Теперь наследник:

```java
class Square extends Rectangle {
    @Override
    void setWidth(int width) {
        this.width = width;
        this.height = width;
    }

    @Override
    void setHeight(int height) {
        this.width = height;
        this.height = height;
    }
}
```

На первый взгляд всё выглядит логично: квадрат — частный случай прямоугольника.

Но теперь представим код:

```java
void resize(Rectangle rectangle) {
    rectangle.setWidth(5);
    rectangle.setHeight(10);

    int result = rectangle.area();
}
```

Для обычного `Rectangle` площадь будет:

```text
5 * 10 = 50
```

Для `Square` получится другое поведение:

```text
setWidth(5)   -> width = 5, height = 5
setHeight(10) -> width = 10, height = 10
area()        -> 100
```

Код работал с `Rectangle`, но сломался логически при подстановке `Square`.

Значит:

```text
Square extends Rectangle нарушает LSP.
```

Важный вывод:

```text
Математическая истина "квадрат является прямоугольником"
не равна ООП-истине "квадрат является корректным наследником Rectangle".
```

В программировании важнее поведение, а не классификация по названию.

## Как исправить Rectangle и Square

Часто решение — не наследование, а более правильная абстракция.

Например:

```java
interface Shape {
    int area();
}
```

И отдельно:

```java
class Rectangle implements Shape {
    // width, height
}

class Square implements Shape {
    // side
}
```

Теперь мы не говорим, что квадрат можно изменять как прямоугольник. Мы говорим только об общем стабильном поведении:

```text
У фигуры можно посчитать площадь.
```

Это хороший подход: делать родителя не по общим полям, а по общему контракту.

## Пример нарушения: Bird и Penguin

Плохая модель:

```java
class Bird {
    void fly() {
        // flying
    }
}

class Penguin extends Bird {
    @Override
    void fly() {
        throw new UnsupportedOperationException();
    }
}
```

Почему плохо?

Потому что `Bird` обещает, что птица умеет летать. Но не все птицы умеют летать.

Лучше разделить сущность и способность:

```java
interface Bird {
    void eat();
}

interface Flying {
    void fly();
}

class Sparrow implements Bird, Flying {
    public void eat() {
        // eating
    }

    public void fly() {
        // flying
    }
}

class Penguin implements Bird {
    public void eat() {
        // eating
    }
}
```

Теперь `Penguin` не обязан реализовывать то, чего не умеет.

Код, которому нужен именно полёт, должен принимать не `Bird`, а `Flying`:

```java
void launch(Flying flyingThing) {
    flyingThing.fly();
}
```

## Связь LSP с полиморфизмом

Полиморфизм говорит:

```text
Можно вызвать один и тот же метод у разных реализаций.
```

LSP добавляет:

```text
Эти реализации должны быть взаимозаменяемы без сюрпризов.
```

Без LSP полиморфизм становится опасным.

Ты вроде пишешь универсальный код через базовый тип, но потом вынужден добавлять проверки:

```java
if (bird instanceof Penguin) {
    // особый случай
}
```

Когда таких проверок становится много, это сигнал: иерархия классов выбрана неудачно.

## Типичные нарушения LSP

### 1. Наследник кидает UnsupportedOperationException

```java
class ReadOnlyList extends SomeList {
    @Override
    void add(Object value) {
        throw new UnsupportedOperationException();
    }
}
```

Если родитель обещает изменяемость, readonly-наследник ломает контракт.

### 2. Наследник сужает допустимые аргументы

```java
class Storage {
    void save(String value) {
        // accepts any non-null string
    }
}

class ShortStorage extends Storage {
    @Override
    void save(String value) {
        if (value.length() > 10) {
            throw new IllegalArgumentException();
        }
    }
}
```

Если базовый класс принимал любые строки, наследник не должен внезапно принимать только короткие.

### 3. Наследник возвращает более слабый результат

Например, родитель обещает отсортированную коллекцию, а наследник возвращает неотсортированную.

Сигнатура та же, компилятор доволен, но контракт нарушен.

### 4. Наследник меняет смысл метода

```java
class Door {
    void open() {
        // door becomes open
    }
}

class LockedDoor extends Door {
    @Override
    void open() {
        // does nothing
    }
}
```

Если `open()` у родителя означает “после вызова дверь открыта”, то пустая реализация у наследника ломает ожидание.

## Альтернативы наследованию

Главная альтернатива — **композиция**.

Вместо:

```text
ElectricCar extends EngineCar
```

часто лучше:

```text
Car has Engine
```

То есть “машина имеет двигатель”, а не “электромобиль является бензиновым автомобилем”.

Ещё полезные подходы:

```text
interface вместо abstract class
разделение интерфейсов по способностям
Strategy pattern
Composition over inheritance
sealed classes, если набор наследников ограничен
```

Если поведение меняется, можно вынести его в стратегию:

```java
interface Movement {
    void move();
}
```

Тогда объект будет не наследоваться от “летающего”, “плавающего” или “ходящего” класса, а содержать нужное поведение внутри.

## Как думать на практике

Когда хочется сделать:

```java
class B extends A {
}
```

стоит спросить себя:

```text
1. B действительно является A по поведению, а не только по названию?
2. Любой код, который работает с A, сможет работать с B без специальных if?
3. B не отключает методы родителя?
4. B не кидает новые неожиданные исключения?
5. B не требует более строгих входных данных?
6. B сохраняет обещания родителя после выполнения методов?
```

Если хотя бы один ответ неприятный, лучше подумать про интерфейс или композицию.

## Связь с Java

В Java LSP проявляется через:

```text
extends
implements
method overriding
abstract classes
interfaces
polymorphism
runtime dispatch
```

Компилятор Java проверяет типовую совместимость, но не может полностью проверить поведенческий контракт.

Java разрешит такой код:

```java
@Override
void fly() {
    throw new UnsupportedOperationException();
}
```

С точки зрения компилятора всё нормально. С точки зрения архитектуры — скорее всего плохо.

Поэтому LSP — это принцип проектирования, который держится на дисциплине разработчика, тестах и ясных контрактах.
