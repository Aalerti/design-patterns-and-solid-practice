# SOLID Practice: Dependency Inversion Principle

Твоя задача - исправить код так, чтобы он лучше соблюдал Dependency Inversion Principle.

> Модули верхнего уровня не должны зависеть от модулей нижнего уровня. Оба должны зависеть от абстракций.

## Что сейчас не так

В `Main.java` есть сервис `PasswordResetService`, который сам создаёт внутри себя объект `SmsGateway`:

```java
private final SmsGateway smsGateway = new SmsGateway();
```

`PasswordResetService` — это модуль верхнего уровня (бизнес-логика).
`SmsGateway` — это модуль нижнего уровня (детали инфраструктуры).

Это нарушение DIP: высокоуровневый класс сам решает, как именно доставлять сообщения.
Если завтра понадобится отправлять код по email или через push-уведомление — придётся менять `PasswordResetService`, хотя его бизнес-логика не изменилась.

## Задание

Исправь структуру так, чтобы:

- `PasswordResetService` не знал о конкретном способе доставки сообщения;
- зависимость от конкретного отправителя передавалась снаружи (через конструктор);
- можно было легко добавить `EmailSender` или `PushSender` без изменения `PasswordResetService`;
- в `main()` явно выбирался нужный способ доставки.

## Подсказка

Введи абстракцию между сервисом и деталями доставки:

```java
interface NotificationSender {
    void send(String recipient, String message);
}
```

Тогда `SmsGateway` будет реализовывать этот интерфейс, а `PasswordResetService` получит его через конструктор — и не будет знать, кто именно за ним стоит.

## Самопроверка

После решения спроси себя:

- Создаёт ли `PasswordResetService` внутри себя конкретный класс отправителя?
- Можно ли передать в `PasswordResetService` другой способ доставки (email, push) без изменения его кода?
- Зависит ли `PasswordResetService` от интерфейса, а не от конкретного класса?
- Явно ли в `main()` выбирается нужный отправитель?
