# D — Dependency Inversion Principle (Принцип инверсии зависимостей)

## 1. Теория «на пальцах»

У принципа есть две формулировки от Роберта Мартина, и их важно держать в голове целиком:

> **Модули верхнего уровня не должны зависеть от модулей нижнего уровня. Оба должны зависеть от абстракций.** **Абстракции не должны зависеть от деталей. Детали должны зависеть от абстракций.**

Чтобы это не звучало как мантра, давай разложим, что вообще такое «верхний» и «нижний» уровень.

**Модуль верхнего уровня** — это бизнес-логика, то, _что_ система делает. Например, класс, который решает «надо оповестить клиента». Он оперирует смыслом, а не техническими деталями.

**Модуль нижнего уровня** — это конкретная реализация, то, _как_ это делается технически. Например, отправка через SMS, через email, через push. Это «руки», а не «мозг».

Теперь главная проблема, которую решает принцип. В наивном коде верхний уровень напрямую создаёт и использует нижний: бизнес-логика внутри себя пишет `new SmsSender()`. И тут возникает жёсткая связь (tight coupling). Что плохого? Во-первых, если ты захочешь поменять SMS на email, тебе придётся лезть в код бизнес-логики и переписывать его — хотя сама логика «оповестить клиента» не изменилась. Во-вторых, такой класс почти невозможно нормально протестировать: ты не можешь подсунуть ему «фейковый» отправитель, потому что он намертво приколочен к настоящему. В-третьих, верхний уровень — самое ценное в системе, ваша уникальная бизнес-логика — оказывается в рабстве у мелкой технической детали. Это переворот ценностей с ног на голову.

Решение: вставить между ними **абстракцию** (интерфейс). Верхний уровень зависит от интерфейса. Нижний уровень этот интерфейс реализует. Теперь оба смотрят на абстракцию, а не друг на друга.

**Откуда слово «инверсия»?** Это ключевой момент, который любят спрашивать. Поток управления (control flow) никуда не девается — верхний уровень всё так же командует нижним во время выполнения. А вот **направление зависимости в исходном коде** переворачивается. Раньше стрелка зависимости шла сверху вниз (верхний → нижний, как и поток управления). Теперь нижний уровень зависит от абстракции (реализует её), а абстракция концептуально «принадлежит» верхнему уровню. То есть стрелка зависимости в коде теперь направлена против потока управления. Вот эта инверсия направления стрелки и дала имя принципу. 🔄

## 2. Схема

Сравни два состояния системы — это самое наглядное во всей теме:

```
БЕЗ DIP (жёсткая связь):

   [ Car ]  ────depends on────►  [ PetrolEngine ]
  (верхний)                         (нижний)

  Поток управления:  ──────────►
  Зависимость кода:  ──────────►   (идут в одну сторону)


С DIP (зависимость инвертирована):

   [ Car ]  ────depends on────►  « Engine »   ◄────implements────  [ PetrolEngine ]
  (верхний)                    (абстракция)                          (нижний)

  Поток управления:  ──────────────────────────────────►
  Зависимость кода:  ──────►            ◄──────────────   (стрелка снизу развернулась!)
```

Алгоритм применения принципа в голове:

1. Найди в классе верхнего уровня место, где он создаёт или жёстко использует конкретный класс нижнего уровня (`new ConcreteThing()`).
2. Выдели из этой конкретики интерфейс — описание _что_ нужно уметь, без _как_.
3. Заставь нижний уровень реализовывать этот интерфейс.
4. Заставь верхний уровень зависеть только от интерфейса, а конкретную реализацию получать извне (через конструктор, например).

## 3. Абстрактный пример

Возьмём отвлечённую машину и двигатель. Сначала — **как делать НЕ надо**:

```java
class PetrolEngine {
    void start() {
        System.out.println("Бензиновый двигатель завёлся");
    }
}

class Car {
    // Жёсткая зависимость: Car прибит гвоздями к бензиновому двигателю
    private final PetrolEngine engine = new PetrolEngine();

    void drive() {
        engine.start();
        System.out.println("Машина поехала");
    }
}
```

Здесь `Car` (верхний уровень, «что делает машина») намертво знает про `PetrolEngine` (нижний уровень, конкретная деталь). Хочешь электромобиль — лезь и переписывай `Car`. Хочешь протестировать `drive()` без реального двигателя — никак.

Теперь **правильная версия** с инверсией:

```java
// Абстракция — принадлежит миру Car, описывает ЧТО нужно, без КАК
interface Engine {
    void start();
}

// Детали зависят от абстракции (реализуют её)
class PetrolEngine implements Engine {
    public void start() {
        System.out.println("Бензиновый двигатель завёлся");
    }
}

class ElectricEngine implements Engine {
    public void start() {
        System.out.println("Электромотор тихо ожил");
    }
}

// Верхний уровень зависит ТОЛЬКО от абстракции
class Car {
    private final Engine engine;

    // Конкретную деталь получаем извне, а не создаём внутри
    Car(Engine engine) {
        this.engine = engine;
    }

    void drive() {
        engine.start();
        System.out.println("Машина поехала");
    }
}
```

Разберём построчно, почему это работает. Поле теперь имеет тип `Engine` — это интерфейс, абстракция, а не конкретный класс. `Car` понятия не имеет, бензиновый там двигатель или электрический; он знает только, что у этой штуки есть метод `start()`. Конкретную реализацию мы передаём через конструктор (`Car(Engine engine)`) — это и есть точка, через которую снаружи решается, какую деталь подставить. Метод `drive()` вызывает `engine.start()`, и благодаря динамической диспетчеризации (помнишь dynamic dispatch из блока 1.6?) в рантайме отработает метод того конкретного двигателя, который реально подставили.

Использование выглядит так:

```java
Car petrolCar = new Car(new PetrolEngine());
Car electricCar = new Car(new ElectricEngine());
```

Обрати внимание на важную вещь: момент `new PetrolEngine()` никуда не исчез — он просто **переехал наружу**, из недр `Car` в код, который собирает систему. Бизнес-логика очистилась от технической детали. А ещё теперь в тесте ты легко подсунешь `new FakeEngine()` и проверишь `Car` в изоляции.

И вот тут — **самая частая ошибка собеседований**. То, что мы передаём `Engine` через конструктор — это **Dependency Injection (внедрение зависимостей)**, отдельная техника. А **DIP** — это сам факт того, что `Car` зависит от интерфейса `Engine`, а не от конкретного класса. DI — это один из способов реализовать DIP, но не сам принцип. Можно соблюдать DIP и без DI (например, доставать реализацию через фабрику), и можно делать DI без настоящей инверсии (внедрять конкретный класс вместо интерфейса). Не путай! ⚠️

## 4. Альтернативы и нюансы

**Кому принадлежит интерфейс?** Это тонкий момент, отличающий «настоящую» инверсию от косметической. Интерфейс `Engine` должен концептуально принадлежать миру `Car` (пакету верхнего уровня), а не миру двигателей. Если интерфейс лежит рядом с реализациями в «нижнем» пакете, ты формально ввёл абстракцию, но направление зависимости по-настоящему не перевернул. В чистой архитектуре интерфейс определяет потребитель (верхний уровень), а поставщик (нижний) под него подстраивается.

**Способы внедрения зависимости** (это уже мостик к блоку 6.4 по Spring): через конструктор (самый предпочтительный — зависимость обязательна и поле можно сделать `final`), через сеттер (для опциональных зависимостей), через поле напрямую (удобно, но плохо тестируется и скрывает зависимости). Запомни связку: constructor injection — золотой стандарт.

**Service Locator как альтернатива DI.** Вместо того чтобы получать зависимость снаружи, объект сам идёт в некий «реестр» и просит нужную реализацию. Это тоже снимает жёсткую связь с конкретным классом, но считается полу-антипаттерном: зависимости становятся неявными, их не видно в сигнатуре конструктора, и тестировать сложнее. DI почти всегда лучше.

**Не переусердствуй.** Очень важная мысль для здорового кода: DIP нужен не для _каждой_ зависимости. Если зависимость стабильная и почти не меняется — например, классы стандартной библиотеки (`String`, `List`, математика) — оборачивать их в свои интерфейсы бессмысленно, это раздувает код. Принцип ценен там, где деталь **может смениться** или где нужна **подмена в тестах**. Плодить интерфейс с единственной реализацией «на всякий случай» — частая ошибка новичков, которая засоряет проект.

**Связь со Spring** (забегаем в блок 6.1, тебе это прямо сейчас актуально). Весь Spring построен вокруг DIP. Когда ты пишешь `@Autowired private NotificationSender sender;`, где `NotificationSender` — интерфейс, ты соблюдаешь DIP, а Spring-контейнер через свой механизм DI подставляет нужную реализацию. Здесь и сходятся три понятия: **IoC** (Inversion of Control — общая идея, что контролем над созданием объектов управляет не твой код, а контейнер), **DI** (конкретная техника внедрения) и **DIP** (принцип проектирования, ради которого всё затевалось).

**Связь с буквой L (LSP).** D и L работают в паре: раз верхний уровень полагается на абстракцию, любая реализация должна быть корректно взаимозаменяемой, не ломая ожиданий. Если подстановка `ElectricEngine` вместо `PetrolEngine` неожиданно ломает поведение `Car` — это уже нарушение Лисков, и оно подрывает всю пользу от DIP.
