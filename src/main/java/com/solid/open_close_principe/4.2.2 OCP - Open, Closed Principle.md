# Блок 4.2 — Open/Closed Principle (буква O в SOLID)

### 1. Теория на пальцах

Принцип сформулировал Бертран Мейер ещё в 1988 году в книге _"Object-Oriented Software Construction"_, и звучит он так: программные сущности — классы, модули, функции — должны быть **открыты для расширения, но закрыты для модификации**. На первый взгляд это похоже на противоречие: как можно одновременно «расширять» и «не трогать»? На самом деле никакого парадокса нет, просто речь идёт о двух разных действиях с кодом.

«Закрыт для модификации» означает, что код, который уже написан, протестирован и работает в проде, в идеале остаётся неприкосновенным, когда появляется новый кейс использования. Причина простая и сугубо практическая: любое изменение существующего файла — это риск **регрессии**, то есть случайно сломать то, что раньше работало правильно. Чем больше людей правят один и тот же кусок логики по разным причинам, тем выше шанс конфликта и побочных эффектов, которые всплывут не сразу, а через две недели в продакшене.

«Открыт для расширения» означает, что добавлять новое поведение всё равно можно и нужно — просто механизм добавления должен быть **через новый код**, а не через правку старого. Под капотом за этим стоит ровно тот механизм, который мы разбирали в теме 4.1 — полиморфизм и абстракция. Если вызывающий код общается не с конкретным классом, а с интерфейсом, то добавление нового варианта поведения превращается в создание нового класса, реализующего этот интерфейс. Старый код, который этот интерфейс использует, физически не открывается заново, а значит, не может случайно пострадать.

Самый узнаваемый антипаттерн, который нарушает OCP — это разрастающаяся цепочка условий по типу или строковому значению:

```
if (type.equals("CARD")) { ... }
else if (type.equals("PAYPAL")) { ... }
else if (type.equals("CRYPTO")) { ... }  // кто-то добавил эту ветку позже
```

Каждый новый способ оплаты требует **редактирования** этого конкретного метода, а значит — риска зацепить уже работающие ветки `CARD` и `PAYPAL`, просто неаккуратно вписав код для `CRYPTO` где-то рядом. В индустрии у этого даже есть отдельное имя — _shotgun surgery_ (хирургия дробовиком): один маленький новый кейс требует «выстрела» правок сразу в нескольких местах кода, разбросанных по системе.

Важная оговорка, которая часто ускользает: OCP не требует, чтобы код был закрыт для **любых вообразимых** изменений вообще. Он говорит о закрытости относительно конкретной, заранее предвиденной **оси изменения** — то есть того направления, в котором система объективно будет расти. Если ты правильно выделил абстракцию, но в реальности оказалось, что сама абстракция была неверной (например, выяснилось, что у каждой формы должна быть ещё и единица измерения, а интерфейс этого не предусматривал), тебе всё равно придётся менять интерфейс и все реализации — и это нормально. OCP защищает не от ошибок в изначальном дизайне, а от того, чтобы рост **числа вариантов** внутри уже верной абстракции требовал трогать существующий код.

### 2. Схема и алгоритм применения

Сначала визуально — как выглядит нарушение и как выглядит соблюдение принципа:

```
НАРУШЕНИЕ OCP
добавление нового варианта = правка существующего метода

   PaymentProcessor.process(type)
            │
            ▼
    if (type == CARD)   {...}   ← существующий код,
    else if (type == PAYPAL) {...}  риск задеть его при правке
    else if (type == CRYPTO) {...}  ← новая ветка дописана ВНУТРИ старого метода


СОБЛЮДЕНИЕ OCP
добавление нового варианта = новый класс, старый код не открывается

         interface PaymentMethod { process(); }
              ▲        ▲         ▲          ▲
              │        │         │           │
       CardPayment PayPalPayment CryptoPayment  NewMethodPayment
                                                  ← просто новый файл,
                                                    ноль правок в старых
```

Алгоритм применения принципа на практике укладывается в четыре шага. Сначала нужно распознать симптом — найти повторяющуюся условную конструкцию (`if-else` или `switch`), которая ветвится по типу объекта или по строковому/числовому коду варианта, и которая, как ты подозреваешь, будет дополняться новыми ветками в будущем. Дальше нужно выделить общий интерфейс или абстрактный класс, описывающий контракт — то есть тот единственный метод (или набор методов), который реально нужен вызывающему коду, без знания деталей конкретной реализации. После этого каждая существующая ветка условия превращается в отдельный класс, реализующий этот интерфейс, а сама условная логика внутри ветки переносится внутрь метода нового класса. И наконец, вызывающий код переписывается так, чтобы он работал только с абстракцией — он получает уже готовый объект нужного типа (через параметр, фабрику или DI-контейнер, о котором мы поговорим в Блоке 6) и просто вызывает метод интерфейса, не зная и не интересуясь, какая конкретно реализация ему подсунута.

### 3. Абстрактный пример

Возьмём отвлечённую сущность `Shape` (геометрическая форма) и задачу — посчитать площадь. Сначала вариант, нарушающий OCP:

```java
class AreaCalculator {
    public double calculateArea(Object shape) {
        if (shape instanceof Circle) {
            Circle c = (Circle) shape;
            return Math.PI * c.getRadius() * c.getRadius();
        } else if (shape instanceof Square) {
            Square s = (Square) shape;
            return s.getSide() * s.getSide();
        }
        // появится Triangle — придётся снова лезть СЮДА и редактировать этот метод
        return 0;
    }
}
```

Метод `calculateArea` знает обо всех конкретных формах сразу — он содержит весь алгоритм расчёта площади для каждого типа внутри себя. Это значит, что добавление треугольника требует открыть этот файл, дописать ещё один `else if`, и понадеяться, что ты ничего не сломал в логике для круга и квадрата, которая стояла рядом.

Теперь версия, соблюдающая принцип:

```java
interface Shape {
    double calculateArea();
}

class Circle implements Shape {
    private final double radius;

    Circle(double radius) {
        this.radius = radius;
    }

    @Override
    public double calculateArea() {
        return Math.PI * radius * radius;
    }
}

class Square implements Shape {
    private final double side;

    Square(double side) {
        this.side = side;
    }

    @Override
    public double calculateArea() {
        return side * side;
    }
}

class AreaCalculator {
    public double calculateArea(Shape shape) {
        return shape.calculateArea();   // делегирование, а не ветвление
    }
}
```

Здесь произошло ключевое смещение ответственности: каждая форма теперь сама знает, как считать собственную площадь, а `AreaCalculator` просто доверяет этому знанию через интерфейс, ничего не зная о конкретных формулах. Когда понадобится `Triangle`, ты пишешь ровно один новый класс:

```java
class Triangle implements Shape {
    private final double base;
    private final double height;

    Triangle(double base, double height) {
        this.base = base;
        this.height = height;
    }

    @Override
    public double calculateArea() {
        return 0.5 * base * height;
    }
}
```

И всё — ни `AreaCalculator`, ни `Circle`, ни `Square` не открываются для редактирования ни на одну строчку. Любой код, который уже использует `AreaCalculator`, продолжает работать без единого изменения, просто теперь умеет принимать ещё и треугольники. Это и есть «закрыт для модификации, открыт для расширения» в чистом виде: расширение системы происходит строго добавлением, а не правкой.

### 4. Альтернативы и важные нюансы

Здесь стоит разобрать сразу несколько практических развилок, потому что слепое применение OCP «всегда и везде» — это так же плохо, как его полное игнорирование.

Первая развилка — это вопрос, действительно ли набор вариантов открыт для роста. Если у тебя объективно **закрытый, фиксированный** набор значений — например, дни недели, или строго оговорённые в контракте три статуса заказа, которые гарантированно не изменятся — оборачивать каждый вариант в отдельный класс с интерфейсом — это лишняя сложность без выгоды, тот самый антипаттерн _speculative generality_ (умозрительная универсальность), о котором мы уже говорили в контексте SRP. В таком случае обычный `switch` по `enum` абсолютно нормален и даже предпочтителен.

Здесь возникает по-настоящему интересный и часто звучащий на сильных собеседованиях нюанс: начиная с Java 17, появились **sealed classes** (тема 1.16 нашего плана) и exhaustive pattern matching в `switch`. Это механизм, который сознательно работает **в противовес** духу OCP — он специально заставляет компилятор требовать от тебя обработки **каждого** варианта sealed-иерархии, и если ты добавишь новый подкласс, не обработав его в каком-то `switch`, код просто не скомпилируется. То есть для закрытых иерархий типов индустрия осознанно выбирает «закрытость для расширения без модификации» как **достоинство**, а не недостаток: ты хочешь, чтобы компилятор гарантированно напомнил тебе обработать новый случай везде, где это нужно, а не понадеялся на полиморфизм, который может «незаметно» проскочить мимо забытого места. Так что OCP — это инструмент для **открытых** множеств вариантов (способы оплаты, форматы экспорта, виды скидок), а sealed + exhaustive switch — инструмент для **закрытых** множеств, где безопаснее, чтобы компилятор принудительно заставлял тебя всё перечислить.

Вторая развилка касается формального паттерна, который реализует OCP практически в учебниковом виде — это **Strategy** (паттерн, который мы разберём отдельно в Блоке 4.5). По сути всё, что мы только что сделали с `Shape`, и есть Strategy: вынесение алгоритма в отдельную иерархию классов, подставляемых через интерфейс. Когда на собеседовании спрашивают «приведи пример OCP в коде», ожидаемый ответ почти всегда — это Strategy либо что-то очень похожее на него по структуре.

Третья развилка — это масштаб архитектуры целиком, а не отдельного класса. То, что мы сделали локально с `AreaCalculator`, в большом приложении превращается в **plugin-архитектуру**: новый функциональный модуль подключается к системе без правки существующего ядра, просто реализуя нужный интерфейс и регистрируясь где-то (через конфигурацию, через DI-контейнер). Когда мы дойдём до Spring во Блоке 6, ты увидишь, что вся идея `@Component` + внедрение зависимостей по интерфейсу — это OCP, поднятый на уровень целого фреймворка: добавить новую реализацию какого-нибудь `PaymentMethod` в Spring-приложении значит просто создать новый класс с аннотацией, а весь остальной код, который зависит от интерфейса `PaymentMethod`, не меняется вообще.

И последний важный момент про цену абстракции. У каждого слоя интерфейсов есть реальная стоимость — код становится сложнее **читать линейно**: вместо того чтобы просто прочитать метод сверху вниз и увидеть всю логику, тебе приходится переходить «куда-то ещё» (через `Ctrl+Click` в IDE на реализацию), чтобы понять, что конкретно произойдёт во время выполнения. Это называется потерей **локальности кода**. Поэтому правило простое и не догматичное: применяй OCP там, где рост числа вариантов реально предвидится и подтверждён опытом (новые способы оплаты, новые экспортные форматы), и не плоди интерфейсы там, где вариативность — чистая гипотеза, которая, возможно, никогда не реализуется.

### 5. Ключевые слова для документации

Для самостоятельного поиска: `Open/Closed Principle`, `Bertrand Meyer SOLID`, `polymorphism vs conditional logic`, `Strategy design pattern`, `shotgun surgery code smell`, `speculative generality anti-pattern`, `sealed classes exhaustive switch Java` (отдельно посмотри JEP 409 на `openjdk.org`, там прямо обсуждается этот конфликт между OCP и exhaustiveness), `plugin architecture software design`.
