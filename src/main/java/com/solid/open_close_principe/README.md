# SOLID Practice: Open/Closed Principle

Твоя задача - исправить код так, чтобы он лучше соблюдал Open/Closed Principle:

> Классы должны быть открыты для расширения, но закрыты для изменения.

## Что сейчас не так

В `NotificationPriceCalculator` цена считается через `switch` по `Channel`.

Если завтра появится новый канал, например `TELEGRAM`, придется:

- менять `enum Channel`;
- менять `switch` в `NotificationPriceCalculator`;
- рисковать сломать уже работающие каналы.

Это нарушение OCP: для расширения поведения приходится изменять существующий рабочий код.

## Задание

Добавь новый канал уведомлений `TELEGRAM` с ценой `0.40` за получателя.

Но сделай это через рефакторинг:

- убери `switch` из `NotificationPriceCalculator`;
- вынеси расчет цены в отдельные классы или стратегии;
- сделай так, чтобы новый канал можно было добавить новым классом, почти не трогая старую логику;
- оставь запуск через `Main`, чтобы было видно цены для `EMAIL`, `SMS`, `PUSH` и `TELEGRAM`.

## Подсказка

Можно начать с интерфейса вроде:

```java
interface NotificationPricePolicy {
    boolean supports(Channel channel);

    double calculate(Notification notification);
}
```

А `NotificationPriceCalculator` может получать список таких политик и выбирать подходящую.

## Самопроверка

После решения спроси себя:

- Если появится `WHATSAPP`, сколько старых классов придется изменить?
- Можно ли добавить новый способ расчета отдельным классом?
- Стал ли `NotificationPriceCalculator` знать меньше деталей про конкретные каналы?
