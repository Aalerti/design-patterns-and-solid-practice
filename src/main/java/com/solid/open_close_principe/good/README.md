# Good Solution: Open/Closed Principle

В этой версии код исправлен так, чтобы он лучше соблюдал Open/Closed Principle.

## Что было не так в bad-версии

В плохом варианте `NotificationPriceCalculator` использовал `switch` по типу канала. Из-за этого при добавлении нового канала уведомлений приходилось менять уже существующий класс калькулятора.

Например, чтобы добавить `TELEGRAM`, нужно было бы открыть старый `switch` и дописать новую ветку. Это нарушает OCP: система расширяется через изменение уже работающего кода.

## Что сделано в good-версии

- Добавлен интерфейс `Notification`, через который калькулятор работает с любым типом уведомления.
- Общие данные уведомления вынесены в абстрактный класс `NotificationData`.
- Каждый конкретный тип уведомления вынесен в отдельный класс: `EmailNotification`, `SMSNotification`, `PushNotification`, `TelegramNotification`.
- `NotificationCalculator` больше не знает о конкретных типах уведомлений и не содержит `switch`.
- Новый тип уведомления можно добавить новым классом, не меняя логику калькулятора.

## Почему это ближе к OCP

Теперь калькулятор зависит от абстракции `Notification`, а не от конкретных каналов. Он использует только методы интерфейса:

```java
notification.getPricePerRecipient() * notification.getRecipientsCount()
```

Если появится новый канал, например `WhatsAppNotification`, нужно создать новый класс, который реализует нужное поведение через наследование от `NotificationData`. Старый `NotificationCalculator` менять не придется.

## Что можно улучшить дальше

В этой учебной версии цена хранится внутри конкретного класса уведомления. Для более сложной бизнес-логики можно вынести расчет цены в отдельные стратегии, например `PricePolicy`, чтобы уведомление отвечало только за данные, а отдельный класс отвечал за правило расчета цены.
